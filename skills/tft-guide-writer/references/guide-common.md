# TFT 格式化攻略通用规则

这是 `tft-guide-writer` 各 workflow 共用的格式化攻略规则。攻略输出结构、元数据、命名、图片边界、写作风格和内容边界统一维护在这里；维护这些规范时优先修改这个文件，workflow 只负责各自的原稿处理流程。

## 工作区结构

- 攻略笔记放在 `D:\ob\JLH\21 TFT`。
- 攻略素材图放在 `D:\ob\JLH\90 附件\TFT`。
- 从 `21 TFT` 下的攻略笔记引用图片时，通常使用 `../90 附件/TFT/<文件名>`。
- 生成图片是临时产物，输出到 `21 TFT/output`，不作为攻略源文件维护。

## 产物类型判断

拿到原稿后，先判断产物类型，再套对应结构规则，不要混用两套格式。

- **攻略笔记**：阵容攻略、强化符文攻略、装备/机制说明、玩法教学、运营思路、站位和变阵建议。
- **版本更新公告**：Patch Notes、版本更新、平衡调整清单，或主体内容按 `Champion Changes`、`Trait Changes`、`Item Changes`、`Augment Changes` 等模块列出数值改动。
- 如果原稿同时包含正文和网页推荐区，只处理当前指定正文；例如版本更新页底部的 Latest/推荐文章列表应丢弃。
- 如果无法确定类型，按主体信息判断：以“怎么玩”为核心走攻略笔记；以“改了什么”为核心走版本更新公告。

## 统一输出规则

以下规则对攻略笔记和版本更新公告都生效。

- 每篇笔记只使用一个一级标题。
- 新增或写回 `21 TFT` 目录中的 Markdown 文件，文件名必须带两位序号前缀，格式为 `99 - 标题.md` 到 `00 - 标题.md`。
- 序号按从新到旧递增排列：越新的笔记序号越小，排序时排在越上面。
- 新增文件优先使用当前目录里最小的可用两位序号；例如目录已有 `63 -` 到 `99 -`，新稿写成 `62 - 标题.md`。
- 不得复用已有序号；不要把最新笔记排到更大的编号上。
- 正文一级标题不写序号；文件名负责排序，正文一级标题负责展示。
- frontmatter 只保留 `tags`，不添加 `cover`、`title`、`source`、`created`、`description`、`author` 等字段。
- frontmatter `tags` 只用于 Obsidian 分类和检索，写真正有检索价值的关键信息。
- 正文倒数第二个非空行固定写 `来源：<来源名称或 URL>`；原文没有来源时写 `来源：`。
- 正文最后一个非空行固定写小红书话题标签，不写 `小红书标签：` 这类前缀。
- 小红书话题标签总数固定为 `10个`，全部使用 `#话题词` 格式并以空格分隔。
- 小红书话题标签固定保留 `#金铲铲新手教学 #金铲铲攻略 #金铲铲之战`，另外 `7个` 优先从 frontmatter `tags` 补足；多于 `7个` 时自行删减，少于 `7个` 时自行补齐。
- 标题保持短句，方便图片化后阅读。
- 只有原稿已经包含图片引用、用户明确提供封面图，或用户明确要求保留/补图时，才放置封面、截图、阵容图、装备图或奖励图。
- 不要因为本地素材目录里存在相关图片，就自动给笔记补封面图或正文图片。
- 如果该章节没有合适素材图，不要用无关图；至少保留二级标题前的语义图标，并在交付回复中说明缺图。
- 只有真正重要的结论才使用引用块。
- 只有表格比文字更清晰时才使用表格；版本更新公告另有禁用表格规则。

## 正文自然化检查

- 输出到 Markdown 文档前，检查正文是否自然、直接、适合中文攻略阅读。
- 只优化正文语言：去掉机械连接、夸张口号、过度铺垫和僵硬句式。
- 不改格式元素：保留标题层级、标题前图标、frontmatter、标签、分页标记、图片引用、粗体术语、列表结构和来源信息。
- 不借语言优化补写新战术内容；只让已有信息更自然、更像人工整理。

## 攻略笔记结构

攻略笔记使用 Obsidian 属性 + 正文字段的双通道格式：

正文写作风格必须同时遵循同目录的 `guide-writing-style.md`。处理攻略笔记时，读取本文件后必须继续读取该文件；该文件只约束攻略笔记正文，不约束版本更新公告。如果结构格式和写作风格发生冲突，优先保持本文件规定的 Markdown 结构，再在结构内按写作风格压缩句子。

```markdown
---
tags:
  - 霞
  - 烬
  - 观星者
  - 鬼索的狂暴之刃
---

# 攻略标题

标签：运营95 / 高难度 / 登顶阵容
封面：

## 💡 概要

...

来源：
#金铲铲新手教学 #金铲铲攻略 #金铲铲之战 #霞 #烬 #观星者 #鬼索的狂暴之刃 #运营95 #高难度 #登顶阵容
```

规则：

- 正文一级标题只写展示用标题，例如文件名 `99 - 霞 烬 努努.md` 对应正文 `# 霞 烬 努努`。
- 一级标题下必须紧跟两行固定字段：`标签：...` 和 `封面：...`。
- `标签：...` 和 `封面：...` 后空一行，再进入第一个二级标题。
- 正文 `标签：` 只用于图片生成和对外展示，数量控制在 `3个` 以内，使用 ` / ` 分隔。
- 正文 `标签：` 应使用面向读者的概括词，例如 `新手`、`高难度`、`冷门阵容`、`稳定上分`、`8级D牌`。
- 正文 `封面：` 只用于图片脚本提取封面；没有用户提供或原稿明确引用的封面图时保持空值 `封面：`。
- 正文主章节使用二级标题，二级标题文本前加语义图标，例如 `## 💡 概要`、`## ⭐ 最终阵容`、`## 🎒 装备`。
- 允许使用三级标题，禁止使用四级及以下标题。三级标题只用于二级模块下的明确小节，例如 `## 🎒 装备选择` 下的 `### 维克托`、`### 俄洛伊`、`### 娜美`；更细的小节标题继续用加粗文本代替。
- 攻略笔记正文禁用有序列表和无序列表，避免影响后续在内容中间插入分页标签。
- 条件、装备优先级、运营节奏和注意事项等内容改用短段落或加粗引导句组织，不要写成列表项。

推荐章节类型：

- 概要 / 阵容概述
- 最终阵容
- 适合开玩的条件
- 运营节奏
- 装备选择
- 强化符文
- 站位 / 变阵
- 注意事项

章节去重与信息归属：

- 推荐章节只是可选归类，不是必填模板。短碎片稿优先少章节，2 到 3 个二级标题可以成立。
- 整理前先把原稿拆成信息单元，例如开玩条件、成型目标、阶段动作、质量目标、变阵方案、装备、强化符文、站位和注意事项。
- 每个信息单元只能写入一个正文位置；可以按内容类型移动分类，但不能在概要、成型羁绊、运营节奏、注意事项等章节中重复改写同一信息。
- 只有原稿提供独立运营链时才写 `运营节奏`。如果只是质量目标或成型目标，例如“3到4阶段需要2星纳尔”“8级D出7木灵族2狙神”，且已在概要或成型章节表达，不再单独拆出运营节奏。
- 不要为了让攻略显得完整而新增总结型章节；无独立信息支撑的复述章节应删除或并入首次出现的位置。

## 版本更新公告结构

版本更新公告以 `21 TFT/65 - 17.4 版本更新公告.md` 为参考模板：

```markdown
---
tags:
  - 版本更新
  - 英雄改动
  - 羁绊改动
  - 装备改动
  - 强化符文
  - 平衡调整
---

<!-- tft-page: 1 -->

# 17.5 版本更新

🟢 加强｜🔴 削弱｜🟡 调整｜🟠 重做｜⚫ 移除

## ⚔️ 英雄

**🟢 凯特琳**

- 瞄准脑袋伤害：`190 / 285 / 555` → `210 / 315 / 585`
- 新星特攻队瞄准脑袋伤害：`76 / 114 / 222` → `84 / 126 / 234`

## 🧩 羁绊

...

来源：TFTtomus
#金铲铲新手教学 #金铲铲攻略 #金铲铲之战 #17.5版本 #版本更新 #英雄改动 #羁绊改动 #装备改动 #强化符文 #平衡调整
```

规则：

- frontmatter `tags` 推荐使用 `版本更新`、`英雄改动`、`羁绊改动`、`装备改动`、`强化符文`、`平衡调整`，再按原稿内容补充具体版本标签，例如 `17.5版本`。
- 文件名写成 `33 - 17.5 版本更新公告.md` 这类形式；正文一级标题写成 `# 17.5 版本更新`，不写文件名里的“公告”，也不写序号。
- 版本更新公告不写攻略笔记专属的 `标签：...` 和 `封面：...` 字段。
- 一级标题后直接放图例：`🟢 加强｜🔴 削弱｜🟡 调整｜🟠 重做｜⚫ 移除`。
- 按原稿内容分组输出二级标题，常用分组为 `## ⚔️ 英雄`、`## 🧩 羁绊`、`## 🎒 装备`、`## 🧪 赛博改动`、`## ✨ 强化符文`。原稿没有的分组不要硬补。
- 每个改动对象使用加粗标题，格式为 `**🟢 英雄或项目名**`、`**🔴 装备名**`、`**🟡 羁绊名**`、`**🟠 项目名**`、`**⚫ 项目名**`。
- 改动明细使用无序列表；数值用反引号包裹，并尽量统一空格格式，例如 `` `190 / 285 / 555` → `210 / 315 / 585` ``。
- 一个对象有多条改动时，保留在同一个加粗标题下，不拆成表格。
- 版本更新公告优先使用短列表，不使用表格，不写概要段落，不补玩法分析。
- 原稿里的 `BUFF` 写作 `🟢 加强`，`NERF` 写作 `🔴 削弱`，`ADJUST` 写作 `🟡 调整`，`REWORK` 写作 `🟠 重做`，`REMOVED` 写作 `⚫ 移除`。
- 小红书话题标签的另外 `7个` 优先使用版本号和 `版本更新`、`英雄改动`、`羁绊改动`、`装备改动`、`强化符文`、`平衡调整`。

## 强调规则

- 加粗用于英雄、装备、羁绊、强化符文等核心名词。
- `<u>...</u>` 用于关键时机、必须执行的动作或高优先级提醒。
- 同一段落中不要反复强调同一个词。
- 二级标题可以使用语义明确的 emoji，但不要整篇重复同一个图标。

## TFT 写法

- 星级：`3星阿卡丽`
- 费用：`4费英雄`
- 等级：`Lv6`、`Lv8`、`Lv9`
- 阶段：`3-2`、`4-1`
- 羁绊：`数字+羁绊名`，例如 `5新星特攻队`
- 选项：用 ` / ` 分隔
- 流程：只有能提升清晰度时才使用 `→`

## 术语规则

- 最终中文名称以 `references/terms/zh_terms.csv` 为准。
- 英文来源术语参考 `references/terms/en_to_zh.csv`。
- 日文来源术语参考 `references/terms/jp_to_zh.csv`。
- 中文 OCR 或语音误识别参考 `references/terms/误识别映射表.csv`。
- 如果术语无法确认，保留原文，并在回复中说明不确定点，不要把不确定标记写进最终攻略正文。

## 内容完整性

- 整理或翻译时，只处理用户当前提供的文本，或用户明确指定的单篇原稿。
- 不要主动读取、参考、关联同目录其他攻略、同题旧稿、历史整理结果、本地图片、素材目录或其他无关笔记。
- 既然用户单独指定了一篇原稿，就把它当作独立新稿处理；不要因为本地已有同题内容，就复用旧稿里的图片、来源、章节或补充信息。
- 保留原文中的策略含义。
- 不编造数值、单位、强化符文或对局克制关系。
- 不根据常识或套路补写原文没有的弈子定位、装备解释、优劣势、注意事项或总结性结论；缺失内容宁可省略。
- 允许按内容类型重新分类，但不要为了让攻略显得完整而新增总结型章节；每个章节都必须能对应到原文中的具体信息。
- 不要把同一组信息改写后重复放进多个标题下。
- 最终正文使用直接口吻，不要出现“原稿提到”“原文说明”“本文建议”这类来源说明字眼。
- 除非原文提供来源或用户要求，否则不要添加来源 URL。
- 无意义空章节可以省略，有实际信息的内容不要丢失。

## 图片生成准备

- 本地图片使用标准 Markdown 图片语法。
- 路径包含空格时使用尖括号：

```markdown
![alt text](<../90 附件/TFT/example image.png>)
```

- 大图前后保留空行。
- 避免复杂 HTML；只允许少量简单 HTML，例如 `<u>...</u>` 和必要的 `<br>`。
- 图片生成脚本会自动忽略 `来源：...` 下一行的小红书话题标签，不会把它渲染进图片。
